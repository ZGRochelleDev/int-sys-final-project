TS-RAG can suffer from redundant retrieval

This is a known problem in:
 - IR (MMR)
 - RAG systems
 - time-series similarity search


1. I researched the weakness of TS-RAG. I found a list of weaknesses:
  - Limited Performance Gains
  - Ambiguous Value vs. Long Context
  - High Inference Costs
  - Long Context
  - Data Quality Dependency **

2. Of this list, I selected the one that looked the most interesting: Data Quality - I am a data engineer by trade.

3. I then searched for "ways to improve data quality of TS-RAG/RAG"

4. I came accross the term "diversity-aware retrieval" as a potential solution.
  - This is when the retrieval system tries to avoid picking results that are too similar to each other.

  Normal TS-RAG:   “What are the most similar past patterns?” - typical top-k retrieval - (similarity only)
  Diversity aware: “What are the most similar past patterns, but also not just copies of each other?” - (MMR to balances similarity and diversity)

  By the word "results", we mean the time-series segments - aka past data.

  For example, a time-series segment might look like:

    10 → 12 → 15 → 14 → 18

  The pattern is:
    going up, slight dip, then going up again

  Another example:
    30 → 28 → 25 → 23 → 20

  The pattern is:
    steadily going down

  When we say that, "MMR avoids redundant results", we mean that "MMR avoids retrieving too many past time-series segments that show basically the same trend or shape: increasing, decreasing, repeating, or spiking behavior."

3. demonstrate what the repo is currently doing

  For each input time-series window, TS-RAG searches a retrieval database and finds the closest matching historical windows.

    Current pattern (match this):
      up, up, down, up

    Original retrieval might return (with this):
      1. up, up, down, up
      2. up, up, down, up
      3. up, up, down, up
      4. up, up, down, slightly up

    Note that these are all very similar. That is useful because they match the current input, but it can also be redundant.
    Hence why we say, The original method may retrieve many examples that say almost the same thing.

4. demonstrate what I propose I alter.

  MMR means Maximal Marginal Relevance.
  For each input time-series window, TS-RAG searches a retrieval database and finds the closest matching historical windows, but also different from the ones I already selected?”

  So MMR tries to balance two things:
    1. Relevance: Is this retrieved pattern useful for the current input?
    2. Diversity: Is this retrieved pattern different from the patterns already chosen?

    For example:

      Current pattern:
        up, up, down, up

      MMR might return (window = 4):
        1. up, up, down, up
        2. up, down, down, up
        3. up, up, flat, up
        4. up, up, down, strong up

      These are still related to the input, but they are not all duplicates.


  Essentially: Original TS-RAG retrieves the closest past patterns, while MMR retrieves patterns that are both close and diverse.

In many real-world RAG systems, retrieving the top-k most similar results often leads to redundant or overlapping information. Industry resources such as Pinecone and LlamaIndex highlight that naive similarity search can return multiple nearly identical results, which reduces the usefulness of the retrieved context. This motivates the need for diversity-aware retrieval methods such as MMR.
